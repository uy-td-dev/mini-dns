mini-dns

A lightweight, fast DNS server in Rust. Authoritative zones, recursive forwarding & a full recursive resolver, encrypted transports (DoT / DoH / DoQ), DNSSEC, AXFR, a domain blocklist, DNS64, Prometheus metrics, and per-client rate limiting.

Features

  • Record types: A, AAAA, CNAME, MX, TXT, NS, SOA, SRV, PTR, CAA
  • Wildcard records (*.example.com) and CNAME chaining (resolved inline in one response)
  • Case-insensitive lookups; correct AA / NXDOMAIN / NODATA handling with SOA in the authority section for negative caching (RFC 2308)
  • EDNS(0) — honours the client UDP payload size; UDP truncation (TC bit) with TCP fallback; name compression on responses
  • Recursive forwarding with a TTL cache (positive and negative/NXDOMAIN caching), single-flight forwarding, and pooled upstream sockets
  • Full recursive resolver mode — resolve iteratively from the root servers (no dependency on 8.8.8.8/1.1.1.1), with delegation caching
  • Conditional forwarding — route queries for specific domains to different upstream resolvers (longest-suffix match)
  • Multi-zone — serve as many authoritative zones as you like, with correct per-zone SOA and AA handling (longest-suffix zone match)
  • TOML config file — declarative config for zones, forwarding, TLS, metrics, blocklist, DNS64, AXFR, and DNSSEC; CLI flags override file values
  • Domain blocklist (RPZ-style) — block ads/malware domains with wildcard support; hot-reloadable
  • DNS64 — synthesize AAAA from A records using a configurable NAT64 prefix (RFC 6147)
  • AXFR zone transfer — with IP-based ACL (CIDR allowlist) for secondaries
  • DNSSEC — DNSKEY/RRSIG/DS/NSEC parse + encode, ED25519 key generation (RFC 8080), and RRset signing with signature verification
  • Encrypted transports: DNS-over-TLS (DoT), DNS-over-HTTPS (DoH over HTTP/1.1 & HTTP/2), and DNS-over-QUIC (DoQ, RFC 9250)
  • Per-client rate limiting to mitigate floods/amplification
  • Graceful shutdown on SIGTERM/Ctrl+C — drains in-flight queries before exit; /healthz endpoint for load balancers
  • Hot zone reload on SIGHUP (no restart needed)
  • Prometheus metrics at /metrics (plain HTTP), plus in-process metrics logged periodically
  • Configurable via CLI flags, environment variables, or TOML; structured logging via tracing

Architecture

mini-dns is built around a single shared ServerState that every transport and worker reads from. State is deliberately lock-free or sharded so that the hot path (answering from the local zone or cache) never contends across cores.

How it works

Every transport (UDP, TCP, DoT, DoH, DoQ) parses raw bytes into a DnsPacket and calls ServerState::resolve (or resolve_local for the UDP fast path). The state holds Arc references to the zones, cache, forwarder, etc., so many workers read concurrently without locking. Any mutation (zone reload, blocklist reload) uses ArcSwap::store — an atomic pointer swap. In-flight queries keep their Arc to the old snapshot (it isn't freed until the last reader finishes); new queries see the new snapshot immediately.

Why this design

  • Arc + ArcSwap instead of Mutex: the zone and blocklist change very rarely (only on SIGHUP) but are read on every query. A Mutex would become a bottleneck; ArcSwap makes reads lock-free.
  • DashMap instead of HashMap + RwLock: the cache and rate limiter are written constantly (on every cache miss). DashMap shards by key hash, so cores hitting different keys never contend on the same lock.
  • Split resolve_local (sync) from forward_ctx (async): the fast path (zone/cache hit) runs inline on the recv worker — no task spawn, no allocation — avoiding scheduler overhead when ~99% of queries are local. Only forwarding (which awaits the network) spawns a task so the recv loop isn't blocked.
  • One shared stream handler for TCP / DoT / DoH / DoQ: after the TLS handshake all of them are just "a stream carrying length-prefixed DNS messages", so they share handle_dns_stream and avoid duplicated code.
  • Dedicated sidecar tasks: reload / signal / metrics live in background tasks and never block the main recv loop — the server keeps answering queries while a new zone is being loaded.

Query flow

This is the exact order of steps from when a DNS query arrives to when a reply is sent. Understanding this flow helps with debugging and configuring the server correctly.

Step-by-step & rationale

  1. Receive query — the UDP recv loop uses SO_REUSEPORT (one socket per core) so the kernel spreads packets across cores via a hash, instead of a single recv loop becoming a bottleneck.
  2. Rate limit — checked before parsing to reject amplification-flood traffic at the door. Drops are silent (no reply) so the server spends no bandwidth on the attacker. Fixed-window + DashMap is used because it's O(1) and needs no sorted structure.
  3. Parse — DnsPacket::from_bytes parses the header, questions and EDNS OPT. On failure we drop + log a warning rather than send SERVFAIL for garbage, so junk can't be turned into a reflection vector.
  4. Blocklist — checked here (after parse, before zone lookup) so blocked names are blocked before forwarding or a zone could answer. A synthesized NXDOMAIN is returned because clients cache negative answers and stop retrying immediately, reducing recurring load for ad/tracker domains.
  5. Zone lookup — ZoneSet::find picks the longest-suffix-matching zone (e.g. corp.example.com wins over example.com). Zones are sorted by origin length descending at load time, so first-match = longest-match.
  6. Authoritative answer — records are looked up in the zone, with CNAME chase confined to the same zone (to guarantee termination). If the name lives in the zone but has no record of the requested type (NODATA), the SOA is included in the authority section so the client can negative-cache per RFC 2308.
  7. Cache lookup — reached only when the name is not in any local zone, the client set RD (Recursion Desired), and the server offers recursion. The cache stores both positive answers and NXDOMAIN; TTL is the min answer TTL, or the SOA minimum (capped at 3600s) for negative answers.
  8. Forward rule match — ForwardRules::match_name does longest-suffix matching to route *.corp.example.com to an internal resolver while the rest goes to the default upstream or the recursive resolver. This enables "split-horizon DNS" from a single server.
  9. Forward or recurse — Forward: MultiForwarder lazily creates one Forwarder per upstream, each with a pool of connected UDP sockets + a semaphore bounding concurrency (default 64) so upstreams aren't flooded. If a UDP reply has the TC (truncated) bit, it automatically retries over TCP.
    Recurse: RecursiveResolver walks from the root hints, caching delegations per-TLD so subsequent queries under .com don't re-hit the root, with QNAME minimization (RFC 9156).
    Both paths go through single-flight: many concurrent identical queries share one leader's upstream call; followers reuse the result, avoiding a "thundering herd" when a cache entry expires.

Performance

  • SO_REUSEPORT — one UDP socket per core so the kernel load-balances ingress across cores instead of funnelling through a single receive loop
  • Fast path — local and cached answers are resolved synchronously and sent inline: no task spawn or per-packet allocation; only forwarded queries spawn a task
  • Thread-per-core — runtime workers are pinned to CPU cores (best-effort), one per core
  • Lock-free / sharded state — zone via arc-swap, cache and rate limiter via DashMap, metrics sharded and cache-line padded
  • Single-flight forwarding — concurrent identical cache misses collapse into one upstream query; upstream sockets are pooled and bounded by a semaphore
Not yet implemented: io_uring + recvmmsg batching and a thread-per-core runtime such as monoio — these are Linux-only and would replace the Tokio runtime; left as future work.

Why this design

  • SO_REUSEPORT: one UDP socket per core; the kernel hashes (src IP, port) to spread packets onto a fixed socket/core. No single "hot" recv loop becomes a bottleneck, and multiple NIC queues can be used. We bind N sockets (with REUSEPORT) instead of one because UdpSocket::recv_from can't run concurrent recvmmsg on a single socket.
  • Fast path inline: resolve_local runs synchronously on the worker thread without spawning a Tokio task. A zone/cache hit simply builds the response and calls socket.send_to right inside the recv context — zero scheduler overhead, zero allocation beyond &mut buf. Only forwarding (which awaits the network) does tokio::spawn so the recv loop stays unblocked.
  • Thread-per-core + pinning: core_affinity::set_for_current pins each worker thread to a specific CPU, avoiding core migration that would thrash caches. Best-effort because the OS may refuse to honour the affinity.
  • Lock-free / sharded: the zone and blocklist are behind ArcSwap — a read is just an atomic pointer load, no lock. The cache, rate limiter and inflight map use DashMap sharded by key hash, so cores touching different keys never contend. Metrics are sharded and cache-line padded so false-sharing doesn't turn the counters into a serialization point.
  • Single-flight: N queries with the same (name, qtype) cache miss collapse onto one leader that calls the upstream; the N-1 followers wait on a watch::channel. This avoids thundering herds when 1000 clients all query the same name just as a cache entry expires. InflightGuard::drop always removes the entry even if the leader task is cancelled, so the DashMap can't leak.

Install & build

Requires a recent stable Rust toolchain. Build in release mode:

git clone https://github.com/uy-td-dev/mini-dns.git
cd mini-dns
cargo build --release

The binary is produced at ./target/release/mini-dns. Prebuilt binaries for Linux, macOS, and Windows are also attached to each GitHub Release.

Quick start

Run with defaults — listens on 127.0.0.1:8888 (UDP and TCP):

./target/release/mini-dns

Query it with dig:

dig @127.0.0.1 -p 8888 example.com A

Docker

An image and docker-compose.yml are provided. The container exposes UDP/TCP 53 by default and mounts your config and zones:

docker compose up --build

Or build and run manually, passing a TOML config:

docker build -t mini-dns .
docker run --rm -p 53:53/udp -p 53:53 \
  -v $(pwd)/config.toml:/etc/mini-dns/config.toml \
  -v $(pwd)/zones:/zones \
  mini-dns --config /etc/mini-dns/config.toml

Zone file

Records are loaded from a zone file (defaults to zones/example.zone, override with --zone / MINI_DNS_ZONE). Each line is one record:

<domain> <ttl> <type> <data>

Example:

example.com.     3600 SOA   ns1.example.com. admin.example.com. 1 7200 3600 1209600 3600
example.com.     3600 NS    ns1.example.com.
example.com.     3600 A     192.0.2.1
example.com.     3600 AAAA  2001:db8::1
www.example.com. 3600 CNAME example.com.
example.com.     3600 MX    10 mail.example.com.
example.com.     3600 TXT   "v=spf1 -all"
*.example.com.   3600 A     192.0.2.9

Blank lines and lines starting with ; are ignored. A malformed line is logged with its line number and skipped, so one bad entry won't stop the rest of the zone from loading.

Recursion & caching

By default, names not found in any local zone are forwarded to the upstream resolver and the answers are cached for their TTL. Both positive and negative (NXDOMAIN) answers are cached. Use --no-recurse to run as a pure authoritative server (returning NXDOMAIN for unknown names).

For a full recursive resolver (resolve iteratively from the root servers, no dependency on 8.8.8.8/1.1.1.1), set recursive = true under [upstream] in the config. Conditional forwarding rules still apply — matched names go to their specific upstream instead of being resolved iteratively.

How it works

Stub forwarding ([upstream].default): the query bytes are sent verbatim to the upstream through a pool of already-connect()ed UDP sockets. On recv the transaction ID is matched so stale datagrams from a previous query are ignored (sockets are reused). If the upstream sets the TC bit, the forwarder opens a TCP connection, re-sends the query, and returns the full response — the client never sees truncation when the upstream actually has the data.

Full recursive (recursive = true): instead of asking one upstream, the server walks the delegation chain itself.

  1. Start from the 13 root hints (a.root-servers.net … m.root-servers.net).
  2. Send the query (or just the minimal suffix for QNAME minimization, RFC 9156) to a root server.
  3. The root returns a delegation: NS records for the TLD + glue A/AAAA. The delegation is cached keyed by the TLD suffix with its original TTL, so the next query under .com doesn't re-hit the root.
  4. Follow the delegation to the TLD server, then to the authoritative server, until an answer or NXDOMAIN is received.
  5. Depth is bounded by MAX_DELEGATION_DEPTH = 20 (RFC 1034) to prevent infinite loops.

Final answers are cached just like forwarded ones (same Cache), so a repeated query doesn't trigger a fresh resolution.

Why support both modes? Stub forwarding is simple, fast and creates little root traffic — suitable for most cases. Recursive mode makes the server independent — it still works when 8.8.8.8/1.1.1.1 are down or filtered, and it prevents upstream resolvers from profiling the user's DNS activity. Trade-off: recursion costs an extra round-trip per delegation step, but delegation caching makes it much cheaper on the second call.

Conditional forwarding

Route queries for specific domains to different upstream resolvers using [[upstream.rules]]. The longest matching suffix wins, so corp.example.com matches before example.com:

[[upstream.rules]]
suffix = "corp.example.com"
upstream = "10.0.0.53:53"

[[upstream.rules]]
suffix = "internal"
upstream = "10.0.0.53:53"

Unmatched names fall through to the default upstream (or to the recursive resolver if recursive = true).

How it works

At load time the rules are sorted by suffix.len() descending. For each query that misses the zone, match_name iterates the sorted rule list; the first rule whose suffix is equal to the name or for which the name ends with .suffix wins. If no rule matches, the default upstream (if any) or the recursive resolver is used.

Why longest-suffix? So a more specific rule beats a general one. For example corp.example.com → internal resolver is not overridden by example.com → public. This also matches how DNS delegation splits zones — the more specific (child) zone is always authoritative before the parent.

Why not regex? Regexes are more powerful but more expensive O(n) per match and need a parser. Longest-suffix is a simple O(n) that's enough for ~99% of use cases (split-horizon, route-internal-only, per-TLD routing) and gives predictable performance.

Multi-zone

Declare as many authoritative zones as you like with multiple [[zone]] blocks. Zones are matched by longest suffix, so a query for www.corp.example.com matches the corp.example.com zone before the example.com zone. Each zone carries its own SOA and authoritative (AA) handling.

[[zone]]
origin = "example.com"
file = "zones/example.zone"

[[zone]]
origin = "internal"
file = "zones/internal.zone"

How it works

All zone files are loaded into a Vec<AuthZone> and sorted by origin.len() descending. For each query, ZoneSet::find(name) iterates until the first zone whose origin is equal to the name or whose origin the name ends with as .origin wins. The matching zone sets the AA bit and carries its own SOA (so negative caching is correct).

Why sort once at load instead of a BST? The number of zones is usually small (tens); a one-time O(n log n) sort plus O(n) linear lookup with branch-friendly string compares beats the complexity and allocations of a BST at this scale.

Why does the per-zone SOA matter? When returning NXDOMAIN / NODATA, the authority section must carry the SOA of the matching zone so the client caches the negative answer with that zone's TTL. Using a single global SOA would give wrong negative TTLs and could block clients from unrelated TLDs.

Blocklist (RPZ-style)

Block ads/malware domains by listing them in a file — one per line, with ; or # for comments and *. wildcard support. Blocked queries receive a synthesized NXDOMAIN. The blocklist is hot-reloadable.

[blocklist]
file = "zones/blocklist.txt"
# zones/blocklist.txt
ads.example.net
*.malware.example.org   # wildcard block
# tracker.foo.         # commented out

How it works

The blocklist is loaded into a lowercased HashSet<String> held behind an ArcSwap. For each query: lowercase the qname, check the exact name in the set, then walk each parent suffix to match wildcards like *.example.com1. On a hit, a synthesized NXDOMAIN is returned immediately on the fast path — no forwarding, no zone lookup.

Why NXDOMAIN instead of REFUSED or 0.0.0.0? Clients cache NXDOMAIN with its negative TTL, so they don't keep retrying — reducing recurring server load. REFUSED may be retried immediately. 0.0.0.0 looks like a valid IP to apps, which then try to connect and fail confusingly. NXDOMAIN ("does not exist") is the cleanest semantic for a block.

1 The wildcard *.example.com blocks sub.example.com but does not block example.com itself (per standard DNS wildcard semantics).

DNS64

When an AAAA query has no native AAAA record but the name has A records, mini-dns synthesizes an AAAA by embedding the IPv4 address in a NAT64 prefix (RFC 6147). The default well-known prefix is 64:ff9b::/96:

[dns64]
prefix = "64:ff9b::"

How it works

Synthesis only triggers for AAAA queries and when the zone has no native AAAA for the name (so real AAAA records are never overwritten). The A records for the same name are read; for each A, the 4 IPv4 bytes are placed into the last 4 bytes of the prefix to build a synthetic AAAA. The TTL equals the A record's TTL, and the final response sets AA since the server is authoritative for the zone.

Why is this needed? IPv6-only networks (e.g. mobile carriers, IoT) can't connect directly to IPv4. A NAT64 gateway needs an AAAA to know where to route; DNS64 produces a synthetic AAAA pointing into the NAT64 prefix, and the gateway translates the packets back to the real IPv4. The whole process is automatic — clients don't need to change anything.

AXFR zone transfer

Allow secondary servers to transfer a zone over TCP with an IP-based ACL. Empty or omitted allow denies all transfers. Both single IPs and CIDR ranges are accepted:

[axfr]
allow = ["10.0.0.0/8", "192.168.1.5"]
# pull a zone from an allowed secondary
dig @ns1.example.com example.com AXFR

How it works

AXFR is only valid over TCP (RFC 5936). When an AXFR query arrives on a stream transport, try_axfr runs before the normal resolution path:

  1. Check the client IP against Acl::is_allowed (CIDR match). On failure, return REFUSED (rcode 5) without leaking any zone data.
  2. Find the zone matching the qname. If none, return NXDOMAIN.
  3. Frame the response: SOA (opening the transfer) → every other record (sorted by owner name for deterministic output) → SOA again (closing the transfer).

Why ACL before zone check? To avoid leaking which zones exist to unauthorized clients. Like the blocklist, the ACL parses CIDRs once at startup (not per query), so lookups are O(1).

DNSSEC

Enable DNSSEC signing to publish DNSKEY records at zone apexes and sign answer RRsets with RRSIG. mini-dns supports DNSKEY/RRSIG/DS/NSEC record types (parse + encode), ED25519 key generation (RFC 8080), and RRset signing with signature verification.

[dnssec]
enabled = true

How it works

At startup an ED25519 keypair is generated and a DNSKEY record is attached to each zone's apex. For each query (unless the client set DO=0 in EDNS):

  1. If it's a DNSKEY query at an apex, the DNSKEY records are injected into the answer, together with an RRSIG over the DNSKEY RRset itself.
  2. For any query with answer records, the records are grouped by (owner, type) into RRsets; each RRset is signed with ED25519 and the resulting RRSIG is appended to the answer.
  3. Inception = now, expiration = now + 30 days, original_ttl = the RRset's minimum TTL (so a validator can compute the canonical form).

Why ED25519? Signing is much faster than RSA-2048 (μs vs ms), signatures are tiny (64 bytes vs 256), and it's a standard (RFC 8080). Trade-off: not every resolver can verify ED25519, but most modern ones (Unbound, BIND 9.11+, Knot) do.

Current limitation: only signing is implemented — NSEC / NSEC3 for denial-of-existence isn't generated, so negative answers carry no signature. This will be added in a later release.

Encrypted transports (DoT / DoH / DoQ)

Enable DNS-over-TLS and/or DNS-over-HTTPS and/or DNS-over-QUIC by giving them an address. If no certificate is provided, a self-signed certificate for localhost is generated at startup (for local testing only):

./target/release/mini-dns \
    --dot-addr 127.0.0.1:8853 \
    --doh-addr 127.0.0.1:8443 \
    --doq-addr 127.0.0.1:8853

DoH exposes /dns-query (RFC 8484) over HTTP/1.1 and HTTP/2, accepting POST with an application/dns-message body or GET ?dns=<base64url>. DoQ runs DNS over QUIC per RFC 9250.

How it works

All three transports share the same DNS logic once decrypted:

  • DoT: TCP + a TLS handshake (rustls, ALPN dot) → a DNS stream with a 2-byte length prefix per message → handle_dns_stream.
  • DoH: TCP + TLS + HTTP (hyper auto-negotiates HTTP/1.1 or HTTP/2, ALPN h2 / http/1.1) → the /dns-query handler unwraps the POST body or ?dns= query param into a DNS message and calls state.resolve. HTTP/2 keep-alive lets many queries be multiplexed over one connection.
  • DoQ: QUIC (quinn) with a rustls config advertising ALPN doq. Each QUIC stream is one query, the handshake is 1 RTT, and the connection is migration-friendly — faster than TCP+TLS.

Why not one shared listener? Because the transports use different protocols (TCP vs QUIC on separate ports, HTTP on its own path), each needs its own bind. Sharing a single port (e.g. 443 for DoH + DoQ) would require ALPN-based demux; we keep them separate for clarity and easier debugging.

Why generate a self-signed cert? So developers can test DoT/DoH/DoQ right away (kdig +tls -k, curl -k) without mkcert/openssl. Production must supply a real certificate via --tls-cert / --tls-key so clients can trust it.

Rate limiting

Limits the number of queries a client IP can send within a time window (default 1s). Enabled via --rate-limit N or [server].rate_limit = N. 0 = unlimited.

How it works

Fixed-window counter per IP in a DashMap<IpAddr, Window>. Each allow(ip) insert-or-updates the entry: if now - started_at ≥ window the counter resets to 1; otherwise count++ and return count ≤ max. A 60-second background task calls cleanup(), which retains only entries whose window hasn't fully elapsed.

Why fixed-window instead of token bucket? Simpler, O(1), and tiny state (~12 bytes per IP), enough to stop floods and amplification. Token bucket is more precise but needs an extra timestamp plus arithmetic per call. Fixed-window is "good enough" for DNS and costs less memory when an attacker spoofs millions of distinct IPs.

Why silent drops? Replying REFUSED/SERVFAIL still spends bandwidth going back to the client (100 bytes × 10k qps = 1 MB/s of useless replies). Dropping without a reply cuts the return path of an amplification attack entirely. Legitimate clients simply retry with backoff.

Hot reload & graceful shutdown

Send SIGHUP to reload the zone file (or all zones from the config file) in place, without dropping in-flight queries or restarting:

kill -HUP $(pgrep mini-dns)

On SIGTERM or Ctrl+C the server drains in-flight queries before exiting. A /healthz endpoint (on the metrics listener) is available for load balancers.

How it works

SIGHUP reload: a background task waits on signal(SignalKind::hangup) and calls state.reload(). The reload re-reads the zone file / entire config (when the path ends in .toml), builds a new ZoneSet, and publishes it via ArcSwap::store. In-flight queries keep an Arc to the old snapshot, so they stay safe — the old Arc isn't freed until the last reader finishes. The blocklist reloads the same way.

SIGTERM shutdown: the signal handler sends true on a watch::channel and sets an AtomicBool shutting_down. Every recv loop / acceptor does a select! on this channel so it returns immediately when the signal fires. run waits up to SHUTDOWN_GRACE = 5s for spawned tasks, then returns, the runtime drops, and the process exits. /healthz returns 503 once shutting_down is true, so load balancers stop sending new queries immediately.

Why not clear the old zone then load? There'd be a window where the server answers nothing. ArcSwap swaps atomically, so there's always one valid snapshot to read until the swap completes — no downtime window.

Why a 5s grace period? It covers ~99% of forwarded queries (the default upstream timeout is 5s). Anything longer is dropped when the runtime shuts down — acceptable so the process doesn't hang forever in CI/ deploys.

Metrics & health

Pass --metrics-addr (or set [metrics] in the config) to expose Prometheus metrics over plain HTTP at /metrics, plus a /healthz endpoint:

./target/release/mini-dns --metrics-addr 127.0.0.1:9153
curl http://127.0.0.1:9153/metrics
curl http://127.0.0.1:9153/healthz

In-process metrics are also logged periodically.

How it works

Counters and gauges are sharded per core (a DashMap or per-worker array with cache-line padding), so inc_* calls never contend on a lock. When /metrics is scraped, the shards are summed and rendered in Prometheus exposition format (text/plain; version=0.0.4). /healthz just checks the AtomicBool shutting_down flag and returns 200 (ok) or 503 (shutting down).

The main exposed metrics:

  • mini_dns_queries_total — total queries received
  • mini_dns_authoritative_total — answered from a local zone
  • mini_dns_forwarded_total — forwarded / recursed
  • mini_dns_cache_hits_total / cache_misses_total
  • mini_dns_coalesced_total — single-flight followers
  • mini_dns_blocked_total — blocked by blocklist
  • mini_dns_rate_limited_total — dropped by rate limiter
  • mini_dns_errors_total — SERVFAIL / parse failures

Why shard metrics? With a single AtomicU64, each inc_total on any core would be an atomic RMW on the same cache line — heavy contention and cache-line ping-pong. Per-core sharding with cache-line padding avoids false-sharing, so an increment is essentially uncontended.

Why plain HTTP for /metrics? Prometheus scraping usually happens over an internal network or pod localhost. TLS on the metrics endpoint adds operational complexity (cert rotation) for little security benefit once a network policy is in place.

Querying

dig @127.0.0.1 -p 8888 example.com A
dig +tcp @127.0.0.1 -p 8888 example.com A   # force TCP
dig @127.0.0.1 -p 8888 google.com A         # forwarded recursively

# DNS-over-TLS (requires a DoT-capable client, e.g. kdig)
kdig +tls @127.0.0.1 -p 8853 example.com A

# DNS-over-HTTPS over HTTP/2 (raw DNS message via curl; -k trusts the self-signed cert)
printf '\x12\x34\x01\x00\x00\x01\x00\x00\x00\x00\x00\x00\x07example\x03com\x00\x00\x01\x00\x01' \
  | curl -sk --http2 -H 'content-type: application/dns-message' \
      --data-binary @- https://127.0.0.1:8443/dns-query | xxd
